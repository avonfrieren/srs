# srs — Contexte & architecture

Mod Everest **compagnon** de [DemoJameson/Celeste.SpeedrunTool](https://github.com/DemoJameson/Celeste.SpeedrunTool) (MIT), frère de `srta` (même philosophie : dépendance + hooks, jamais de fork). But : importer la practice sheet communautaire (Google Sheets → CSV local) et colorer le temps final d'un segment du room timer selon les paliers de la sheet.
Roadmap dans `PLAN.md`. Le `README.md` sert de **changelog + instructions d'installation** — le mettre à jour à chaque feature.

## Build

- .NET 8 (`net8.0`), projet unique `srs.csproj`. Sur cette machine : `~/.dotnet/dotnet build -p:CelestePrefix="$HOME/.steam/steam/steamapps/common/Celeste"`.
- Références jeu résolues via `CelestePrefix` (auto-détecté si le repo est cloné dans `<Celeste>/Mods/xxx/`).
- **`SpeedrunTool.dll` extraite automatiquement du `SpeedrunTool.zip` officiel installé** (cible `ExtractSpeedrunToolDll`), épinglé sur **v3.27.16** dans `everest.yaml`. Pas de Publicizer pour l'instant (rien d'interne utilisé) — à ajouter comme dans srta si la phase 4 l'exige.
- Cible `OutputAsModStructure` : génère `build/` (DLL + PDB + `everest.yaml` + `Dialog/`) → à zipper en `srs.zip` dans `<Celeste>/Mods/` (comme `srta.zip`) ou copier en dossier `Mods/srs/`.

## Architecture

- `Source/SrsModule.cs` — `EverestModule` ; `Load()`/`Unload()` délèguent aux classes statiques ; `CreateModMenuSection` ajoute les entrées manuelles (bouton de mise à jour, plus tard les listes de sélection de segment — le menu auto-généré d'Everest ne sait pas faire de listes dynamiques).
- `Source/SrsSettings.cs` — settings Everest. `SheetUrl` (`[SettingIgnore]` : pas dans le menu, éditable dans `Saves/modsettings-srs.celeste`) : URL d'édition complète de la sheet, id/gid extraits par regex ; l'ancienne URL par défaut (onglet IL, `LegacySheetUrl`) est migrée vers la nouvelle dans `SrsModule.LoadSettings`. `SelectedChapter`/`SelectedCheckpoint` : sélection persistée par noms (le couple est nécessaire, les noms de checkpoints se répètent entre chapitres).
- `Source/SheetData.cs` — **pur, sans dépendance au jeu** (parseur testable) : parseur CSV RFC 4180 (guillemets, `""`, `\r\n`), parseur de temps `[hh:][mm:]ss[.fff]` (culture invariante), modèle `SheetData` → `SheetBlock` (une ligne d'en-tête = un bloc, détectée par `Hidden`+`WR` en colonnes 1-2 **ou** 2-3 → `TierStart` ; `HasCheckpoints` si la colonne 1 s'appelle `Checkpoint`) → `SheetSegment` (`Chapter` + `Name` ; le nom de chapitre n'est présent que sur la 1re ligne du groupe — cellules fusionnées — donc reporté sur les suivantes ; temps `TimeSpan?` alignés sur les colonnes depuis `TierStart`, null si cellule vide/illisible). `SheetData.CheckpointBlock` = premier bloc à checkpoints, fallback premier bloc (CSV chapitre-seuls : chaque chapitre = son unique checkpoint). Ne lève jamais d'exception sur contenu malformé. **Normalisation post-parse (`Normalize`, appelée par `Parse`** — les noms bruts de la sheet ne sont donc jamais visibles ailleurs ; passe séparée car le pliage exige le bloc complet) : lignes `Wake Up` supprimées (animation, pas un checkpoint) ; chapitres `6a` + `6a Route` + `6b Route` **pliés en un seul `6a/b`** (comme le `5a/b` de la sheet — règle : les chapitres `<X>a/b Route` déclarent les côtés, un chapitre nu nommé comme un côté rejoint le pli), chaque segment plié gardant son `Side` (`"6a"`/`"6b"`, null ailleurs — servira au mapping vers `AreaKey` en phase 4) ; écho du chapitre retiré des noms (`1a Start` → `Start` ; dans les chapitres pliés les préfixes de côté `5a Start`/`6b Start` ne matchent pas `<X>a/b` et survivent) ; noms partagés par les deux côtés d'un pli re-préfixés (`6a Rock Bottom`/`6b Rock Bottom`). `SheetSegment.Chapter`/`Name` mutables pour ça.
- `Source/SheetImporter.cs` — téléchargement (HttpClient statique, timeout 20 s, **async via `Task.Run`, jamais sur le thread de jeu, jamais au lancement**), détection sheet privée (réponse HTML au lieu de CSV), cache `Saves/srs/sheet.csv` (`UserIO.SavePath`, écriture atomique via `.tmp` + `File.Move`), rechargé dans `Load()` ⇒ hors-ligne OK. Le cache **est** aussi le fallback manuel : un CSV déposé à la main y est relu tel quel. Entrées de menu : bouton (label muté depuis le thread worker — sûr, les items relisent leur label chaque frame) + `SubHeader` de statut. Logs sous le tag `srs`.
- `Source/SegmentSelector.cs` — sélection du checkpoint à comparer (phase 4) : `Current` (résolu par `(SelectedChapter, SelectedCheckpoint)` dans `CheckpointBlock`, null si introuvable) ; deux `TextMenu.Slider` dépendants (changer de chapitre vide et re-remplit `Values` du slider checkpoint — liste mutable, pas de reconstruction du menu ; le menu est de toute façon reconstruit à chaque ouverture, donc les données fraîchement importées apparaissent à la réouverture). À l'ouverture, les settings sont resynchronisés sur ce qui est affiché (sélection périmée ⇒ redevient valide). Chaque changement de sélection (et la resynchro d'ouverture, seulement si elle a bougé la sélection) pousse le nombre de rooms du checkpoint dans le « Number of Rooms » de SpeedrunTool via `RoomCounts.Apply`.
- `Source/SegmentAutoDetect.cs` — phase 4bis : quand `AutoDetect` (défaut on), le checkpoint joué pilote la sélection. Chapitre depuis `Session.Area` (table `ChapterMap` : ID+Mode → chapitre sheet + côté des chapitres pliés), checkpoint = dernière room de checkpoint entrée (`Everest.Events.Level.OnEnter` initialise depuis `Session.StartCheckpoint`, null = « Start » ; `OnTransitionTo` + `LevelData.HasCheckpoint` met à jour), room → nom jeu via `CheckpointData` avec `Dialog.Clean` **forcé en anglais** (clés traduites), nom jeu → nom sheet via `CheckpointMap` **hardcodée, sans normalisation** (décision propriétaire 2026-07-18 : la sheet est un prototype, sa successeure aura son propre parseur). La room trackée est enregistrée côté save states ; la sélection est réappliquée chaque frame (couvre les loads de savestate, sans callback) et pousse `RoomCounts.Apply` à chaque bascule. Chargé **après** `TierComparison` : son hook `Level.Update` doit rester externe pour que la capture d'une complétion précède le déplacement de sélection. `Hollows Tape` et la route 5A complète ne sont jamais auto-sélectionnés (absents/ambigus) — la sélection ne bouge pas.
- `Source/RoomCounts.cs` — table statique rooms-par-checkpoint (données communautaires saisies à la main, clé = `(Chapter, Name)` normalisés). `TargetFor` renvoie 99 (`Unknown`, max du slider SpeedrunTool) pour les checkpoints sans donnée (« ? » : finaux de chapitre, Granny, Hollows Tape) — le timer ne se termine alors que par la fin réelle du run (fin de chapitre, cassette, drapeau). `Apply` écrit `SpeedrunToolSettings.Instance.NumberOfRooms`.
- `Source/TierComparison.cs` — phase 4 : hook `Level.Update` (après `orig`) lit `RoomTimerIsCompleted()`/`GetRoomTime()` (ModInterop, jamais le Publicizer) ; sur le front de complétion, capture le temps **seulement si `NumberOfRooms` == `RoomCounts.TargetFor(sélection)`** (sinon = practice partielle, pas de palier) — `GetRoomTime()` continue de courir après complétion, seul l'affichage de SpeedrunTool gèle. Palier recalculé chaque frame depuis le temps capturé ; `wasCompleted`/`hasCapture`/`capturedTicks` enregistrés via l'interop `SpeedrunTool.SaveLoad`. Rendu après `orig` de `SpeedrunTimerDisplay.Render` (masqué si timer Off), décalé d'une ligne si srta est chargé ; couleurs = propriétés statiques de `Color` résolues par réflexion depuis les noms de colonnes. Hotkey `ToggleShowTier` (phase 5) gérée dans le même hook, popup on/off via `PopupMessageUtils` de SpeedrunTool (classe publique, pas de Publicizer).
- `Dialog/English.txt` + `French.txt` — `MODOPTIONS_SRS_TITLE` (titre du menu auto) et clés `SRS_*` (entrées manuelles). Pas de placeholders `{0}` dans les dialogs (Celeste peut interpréter les accolades) : composer en code.

## Données de la sheet (audit 2026-07-18, complété phase 3)

- Onglet importé : « **Celeste Any% Standards CP's** », `gid=276158492`. Export sans compte : `https://docs.google.com/spreadsheets/d/<id>/export?format=csv&gid=<gid>` (sheet « toute personne disposant du lien »).
- Tous les gids (extraits du bootstrap JS de la page `/edit` — pattern `21350203,"[n,0,\"<gid>\",[{\"1\":[[0,0,\"<nom>\"` ; ni `gviz` par nom ni `htmlview`/`mobilebasic` ne marchent, et `gviz` perd des cellules issues de formules) : About `656835588`, 6a CP's `775655909`, 6a IL's `2130893137`, 6b CP's `1763654438`, 6b IL's `959118373`, **Standards CP's `276158492`**, Standards IL's `639470957` (ancien onglet importé, v0.1.0), Fullgame Targets `1345962224`, FileTimes `509878803`.
- En-têtes CP's : `Chapter, Checkpoint, Hidden, WR, Gold, Pink, Purple 1-3, Indigo 1-3, Blue 1-3, Cyan 1-3, Green 1-3, Olive 1-3, Yellow 1-3, Orange 1-3, Red 1-3, Unranked` — **les couleurs sont les noms de colonnes**, les cellules colorées de la sheet ne sont que cosmétiques pour nous. Onglet IL : idem sans la colonne `Checkpoint`.
- Bloc 1 (`Chapter`) : 41 checkpoints en 10 groupes — Prologue (Granny), 1a (1a Start, Crossing, Chasm), 2a (…, Wake Up, Awake), 3a, 4a, `5a/b` (5a Start … Mix Master, route 5a+5b entière), `6a` (6a Start, Lake — commun aux deux routes), `6a Route` (Hollows → Resolution), `6b Route` (Hollows Tape → Reprieve), 7a (7a Start, 500m … 3000m). Le chapitre n'est rempli que sur la 1re ligne du groupe (cellules fusionnées). Bloc 2 (« Chapter Times », colonne Checkpoint vide) : Prologue … 7a + `5a+b`, `6a/6a+b`, `Filetime Buffer`, `Sob`.
- `Unranked` : colonne sans valeurs (palier « au-delà de Red 3 »). `Hidden` : `0:00.000` partout sauf les lignes `Wake Up` (vide). Les lignes `Wake Up` ont des seuils constants par plage (2.55/3.247…).
- Sémantique de comparaison (phase 4) : premier seuil ≥ temps réalisé ⇒ palier atteint.

## Repères SpeedrunTool (pour les phases 3-4)

- ModInterop `SpeedrunTool.RoomTimer` : `RoomTimerIsCompleted()`, `GetRoomTime()` — suffisant pour comparer un temps final ; préférer l'interop au Publicizer (stable entre releases ; DemoJameson accepte des ajouts sur demande Discord).
- ModInterop `SpeedrunTool.SaveLoad` : tout état statique muté en gameplay doit y être enregistré (`RegisterStaticTypes`) sous peine de desync des save states.
- HUD : dessiner après `orig` de `On.Celeste.SpeedrunTimerDisplay.Render` (srs charge après SpeedrunTool ⇒ ses hooks enveloppent les siens) — voir `RoomDeltas.cs` de srta pour le modèle.
